DOMUS FLUTTER - PLAN CONSTRUIBLE EN ORDEN

Objetivo
Este documento es una ruta de construccion para una app Flutter real. No es una idea general. Cada etapa dice que archivos crear, que debe contener cada archivo y cuando se puede considerar terminada.

Documentos base que siempre se pasan junto a este plan
- .agents/domus_api_endpoints.txt
- .agents/flutter_app_docs/18_api_response_contract_ideal.txt
- El TXT especifico de la pantalla que toque.

Consistencia obligatoria con API real
- La API real ya existe hoy. No escribir el plan como si faltaran endpoints.
- No inventar endpoints nuevos en Flutter.
- No usar rutas distintas a las de .agents/domus_api_endpoints.txt.
- Si Laravel cambia una ruta, primero actualizar .agents/domus_api_endpoints.txt y despues este plan si aplica.
- Si este plan menciona una ruta, debe existir en routes/api.php.
- Si .agents/domus_api_endpoints.txt menciona una respuesta, debe coincidir con el controlador real de Laravel.
- Flutter debe construirse contra la API real desde el momento en que se implementa cada feature.
- Las pantallas temporales son validas solo al construir menus y navegacion inicial. Sirven para evitar fallos mientras el modulo todavia no se implementa.
- Una pantalla temporal debe decir claramente que funcion ira ahi, por ejemplo: "Aqui estara Retirar dinero".
- Una pantalla temporal no cuenta como modulo terminado.
- Cuando una etapa implementa un modulo, debe reemplazar la pantalla temporal por la pantalla real conectada a la API real.

Reglas para datos vacios, null y empty states
- Si una lista viene vacia o null, tratarla como lista vacia.
- Si un total numerico viene null porque aun no hay movimientos, mostrar 0.
- Si un monto viene null, mostrar $0.00 usando MoneyText.
- Si un nivel viene null, mostrar "Sin nivel".
- Si una imagen de insignia viene null, mostrar imagen por defecto.
- Si una fecha viene null, mostrar "Sin fecha" o esconder la fila si no aporta contexto.
- Si no hay movimientos recientes, mostrar "Aun no hay movimientos recientes".
- Si no hay prestamos, mostrar deuda 0 y "Todavia no hay prestamos".
- Si no hay usuarios familiares, mostrar "Todavia no has creado usuarios".
- Si no hay tareas, mostrar "No tienes tareas creadas" en padre y "No hay tareas disponibles" en hijo.
- Si no hay recompensas, insignias o canjes, mostrar empty state especifico de esa seccion.
- No mostrar error cuando la API responde correctamente con lista vacia. Eso es estado vacio, no fallo.
- Mostrar ErrorView solo cuando falle la peticion o Laravel devuelva error.

Reglas obligatorias de calidad
- No crear una sola pantalla gigante para padre e hijo.
- No crear un unico root que arme toda la app dinamicamente.
- No poner toda la UI de una pantalla en un solo metodo enorme.
- No poner llamadas API directas dentro de widgets visuales.
- No poner reglas sensibles de dinero, puntos, prestamos o permisos en Flutter.
- No hardcodear colores en cada pantalla; usar AppColors y AppTheme.
- No mezclar responsabilidades: paginas, widgets, repositories y modelos deben estar separados.
- No ahorrar archivos si eso vuelve dificil entender el codigo.
- No exagerar creando un archivo para cada linea; separar cuando tenga sentido real.
- Si una pantalla es diferente para padre e hijo, crear dos pantallas separadas.
- Si una pantalla comparte una tarjeta pequena con otra pantalla, crear widget compartido solo si realmente se reutiliza.
- Cada feature debe tener data, domain y presentation.
- Cada feature debe conectarse a los endpoints reales indicados en su documento.
- Cada accion POST debe usar primero la respuesta de Laravel para actualizar UI.
- Cada pantalla debe tener loading, error, empty state y feedback de exito.

Regla de trazabilidad
Para trabajar una pantalla siempre usar estos documentos en este orden:
1. .agents/flutter_build_order_plan.txt para saber en que etapa va.
2. .agents/domus_api_endpoints.txt para saber rutas, body y respuesta.
3. .agents/flutter_app_docs/18_api_response_contract_ideal.txt para saber como actualizar UI con la respuesta.
4. El TXT de la pantalla, por ejemplo 05_parent_loans.txt o 13_child_tasks.txt.

Regla de ubicacion de codigo
- API de un modulo va en features/modulo/data.
- Modelos del modulo van en features/modulo/domain.
- Pantallas y widgets del modulo van en features/modulo/presentation.
- Widgets usados por varios modulos van en core/widgets.
- Tema, colores y estilos globales van en core/theme.
- Token y sesion global van en core/auth y features/auth.
- Rutas globales van en app/router.

Ejemplo de mala practica prohibida
- Una sola LoansPage con if role en casi todo el archivo.
- Un solo app_state.dart con todos los datos de toda la app.
- Un widget que llama Dio directamente desde build().
- Un archivo de 1500 lineas que mezcla formulario, API, modelos y estilos.
- Una pantalla que recalcula saldo despues de pagar en vez de usar new_balance_display.

Ejemplo de practica aceptable
- ParentLoansPage usa ParentLoansRepository.
- ChildLoansPage usa ChildLoansRepository.
- Ambas pueden compartir LoanCard si visualmente es casi igual.
- El pago de cuota usa la respuesta: loan, payment, new_balance_display, unlocked_badges y domus_points.
- Despues de aplicar esa respuesta, se puede refrescar /api/loans para confirmar.

Dependencias recomendadas en pubspec.yaml
- flutter_riverpod para estado.
- dio para HTTP.
- flutter_secure_storage para guardar token.
- go_router para rutas.
- intl para formato de fechas si hace falta.

No fijar versiones en este TXT. Que Flutter instale versiones compatibles con el proyecto.

Crear lib/core/theme/app_theme.dart:

```dart
import 'package:flutter/material.dart';
import 'app_colors.dart';

abstract final class AppTheme {
  static ThemeData light() {
    final colorScheme = ColorScheme.fromSeed(
      seedColor: AppColors.blue900,
      primary: AppColors.blue900,
      secondary: AppColors.gold500,
      surface: AppColors.surface,
      error: AppColors.errorText,
      brightness: Brightness.light,
    );

    return ThemeData(
      useMaterial3: true,
      colorScheme: colorScheme,
      scaffoldBackgroundColor: AppColors.appBackground,
      fontFamily: 'Segoe UI',
      appBarTheme: const AppBarTheme(
        backgroundColor: AppColors.blue900,
        foregroundColor: Colors.white,
        centerTitle: false,
      ),
      cardTheme: CardThemeData(
        color: AppColors.surface,
        elevation: 0,
        shape: RoundedRectangleBorder(
          borderRadius: BorderRadius.circular(16),
          side: const BorderSide(color: AppColors.border),
        ),
      ),
      filledButtonTheme: FilledButtonThemeData(
        style: FilledButton.styleFrom(
          backgroundColor: AppColors.blue900,
          foregroundColor: Colors.white,
          minimumSize: const Size.fromHeight(48),
          shape: RoundedRectangleBorder(borderRadius: BorderRadius.circular(12)),
        ),
      ),
      inputDecorationTheme: InputDecorationTheme(
        filled: true,
        fillColor: Colors.white,
        border: OutlineInputBorder(borderRadius: BorderRadius.circular(12)),
        focusedBorder: OutlineInputBorder(
          borderRadius: BorderRadius.circular(12),
          borderSide: const BorderSide(color: AppColors.blue700, width: 2),
        ),
      ),
    );
  }
}
```

class ApiException implements Exception {
  ApiException({
    required this.message,
    this.statusCode,
    this.errors,
  });

  final String message;
  final int? statusCode;
  final Map<String, dynamic>? errors;

  @override
  String toString() => message;
